# Course Correction - amogus v2.0 (Tactical Overhaul)

## 1. Impact Analysis

The current implementation (v0.1-ALPHA) is a proof-of-concept that handles basic quest flows but lacks the industrial structure required for real-world IRL use. The introduction of "Batches" and "Admin Roles" represents a major architectural shift.

### Component Impact

- **Database (Redis)**: Schema must be updated to support Batches and Assignments.
- **Frontend Logic**: Transition from "Duration-based draw" to "Assignment-based lookup".
- **Admin App**: New subdomain/route for authenticated Batch management.
- **Quest Components**: Simplified flow for Impostors and Camera-first scanning for all.

### Risk Assessment

- **Breaking Change**: Existing games will be invalidated.
- **Complexity**: Real-time lobby sync requires robust communication (SSE or Polling).
- **Usability**: QR Code export must be user-friendly (PDF with labels).

## 2. Proposed Changes

### Tactical Adjustments

- **PRD/Architecture**: Updated to support Batch/Quest/Content decoupling.
- **Data Layer**: Migration from `@vercel/kv` to standard `redis` client (partially done, needs hardening).
- **Quest System**: Implementation of a dynamic mapper that picks from a JSON library based on Quest format/type.

### Refined Requirements

- **Admin**: Authenticated access, Batch creation, QR PDF Export.
- **Player**: Short-code login, Camera scan, Assignment list, Self-elimination.
- **Impostor**: Invisible auto-success, Credible dashboard.

## What Didn't Go Well

### Recurring Code Review Patterns

1. **State Sync Issues (3/3 stories)**
   - H2 in Story 2.1: Non-atomic GET+SET
   - H2 in Story 2.2: chooseRole didn't sync gameState.players
   - H3 in Story 2.3: Race condition in chooseRole
   - **Root Cause:** Zustand store not consistently synchronized with server responses
   - **Impact:** Rework in every story

2. **Accessibility Gaps (2/3 stories)**
   - M1 in Story 2.2: No prefers-reduced-motion
   - H2 + H5 in Story 2.3: Missing keyboard navigation, no touch target validation
   - **Root Cause:** Accessibility not in dev mental checklist
   - **Impact:** Additional fixes after review

3. **Error Recovery Incomplete (2/3 stories)**
   - H1 in Story 2.1: Error ejected user from lobby
   - M3 in Story 2.2: Retry button didn't actually retry
   - **Root Cause:** Error paths tested less thoroughly than happy paths
   - **Impact:** UX degradation in error scenarios

4. **Increasing Review Issue Count (7 → 8 → 12)**
   - Trend: more issues found per story as complexity grew
   - All issues were fixed, but represents growing rework
   - **Root Cause:** No systematic pre-review checklist

---

## Key Insights & Lessons Learned

### Technical Insights

1. **Always Sync Zustand After Server Response**
   - Every server action that mutates state must update the local store with the response
   - Don't assume local state matches server state after mutation

2. **Accessibility Is Not Optional**
   - prefers-reduced-motion, keyboard navigation, ARIA labels must be checked for every interactive component
   - Build it in from the start, not as a review fix

3. **Composable Components Pay Off**
   - The GameHome extraction in Story 2.3 created reusable building blocks
   - ScanButton is ready for Epic 3 with zero changes needed

### Process Insights

4. **Anticipate Recurring Patterns**
   - A dev checklist covering known patterns prevents repeat issues
   - 27 review issues could have been significantly reduced

5. **Systematic Test Gates Prevent Rework**

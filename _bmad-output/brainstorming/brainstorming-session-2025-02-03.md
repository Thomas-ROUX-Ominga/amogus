## Quest Content – Taxonomy & Minigame Catalogue

**Duration brackets (agreed):**
- **Courte** : < 20 s
- **Moyenne** : 20–40 s
- **Longue** : 40–60 s (max ~60 s)

### Quêtes par durée / difficulté

**Courtes (< 20 s)**  
- Vrai / Faux (une affirmation → clic)  
- Question culture G très simple (réponse courte, une tentative ou 2)  
- Question numérique simple (un chiffre connu, ex. taille Tour Eiffel)  
- Un seul bon choix parmi 2 (A ou B)

**Moyennes (20–40 s)**  
- QCM (3–4 réponses à lire et choisir)  
- Slider / curseur (réglage + validation, ex. année ou pourcentage)  
- Memory 4–6 paires  
- Relier 2–3 fils (couleurs)  
- Code à 4–6 chiffres (saisie)  
- Séquencer 3–4 éléments (ordre chronologique ou logique)  
- Question culture G avec 2–3 essais ou 2e question en cas d’erreur  
- Tap au bon moment (un seul timing)
- Mini-bac : l’app donne 1 lettre, le joueur remplit 3–4 catégories (prénom, star, métier, légume… commençant par cette lettre)

**Longues (40–60 s)**  
- Parcours A→B : scan QR lieu A → indice/nom lieu B → déplacement IRL → scan QR lieu B  
- Memory 8+ paires  
- Puzzle simple (taquin 3×3, ou 4–6 pièces à assembler)  
- Drag & drop tri (plusieurs éléments vers bonnes catégories / poubelles)  
- Simon (répéter une séquence de 4–6 couleurs / sons)  
- Ordre chronologique 5–6 éléments  
- Mini-jeu “éviter obstacles” ou réaction (plusieurs taps)  
- Série de 2–3 questions (culture G ou chiffres) pour une seule quête
- Mini-bac : 1 lettre + 5–6 catégories (prénom, star, métier, légume, ville, animal…), validation par catégorie

- **Quêtes** et **Design** sont le cœur “produit” : ce que le joueur voit et fait à l’écran.

- **Tech** : le MVP (countdown) prouve le flux QR → app ; la prochaine étape logique est “QR → quête” puis “room + rôles”.

## Décisions produit – Rooms & Parcours quête

*(Suite Phase 3 SCAMPER – choix retenus)*

### Rooms

- **Gardé :** lien/code d’invitation, lobby (joueurs connectés), admin lance la partie, pas de compte obligatoire (rejoindre par lien, pseudo à la volée), expiration possible de la room.
- **Écarté :** template / duplication de room pour une prochaine soirée.

**Pools de quêtes :**
- **3 pools** : un par catégorie de durée → `short`, `medium`, `long`.
- **QR → URL :** le scan mène vers  
  **`/game/[id]/quest?duration=short`** (ou `medium` / `long`).  
  L’app affiche **une quête aléatoire** tirée du pool correspondant à `duration`.

### Parcours quête (flux joueur)

1. **Scan QR** → ouvre une quête (page quête, selon `duration` = pool short/medium/long).
2. **Quête terminée** (validation minijeu / réponse / etc.) → **redirection vers la page d’accueil de la partie en cours** (home de la game), pas de reste sur la quête.
3. **V2 (optionnel) :** système de buzzer dans l’app (à préciser plus tard).

## Phase 4 – Plan d’action (Decision Tree)

**Objectif :** Transformer les idées et décisions en étapes implémentables.

### Priorités retenues

| Priorité | Domaine | Justification |
|----------|--------|----------------|
| 1 | Pools + route quête | Fondation : `/game/[id]/quest?duration=short\|medium\|long` + tirage aléatoire dans le bon pool. |
| 2 | Flux quête → home | Fin de quête → redirection vers page d’accueil de la partie. |
| 3 | 2–3 types de quêtes | Valider le flux avec Vrai/Faux, QCM, + 1 minijeu (ex. Memory ou mini-bac). |
| 4 | Room + lobby | Créer partie, lien d’invitation, liste des joueurs, admin lance. |
| 5 | Déploiement Vercel | App en ligne pour les amis. |

### Arbre de décision / ordre de mise en place

```
1. Données & routes
   ├── Modèle "game" (id, statut, joueurs, admin)
   ├── 3 pools de quêtes (short, medium, long) – données ou seed
   └── Route GET /game/[id]/quest?duration=short|medium|long → affiche 1 quête aléatoire du pool

2. Page "home" de la partie
   ├── Route /game/[id] = page d’accueil de la partie en cours
   └── Liens / boutons selon rôle (crewmate : voir quêtes / stats, impostor : …)

3. Flux quête
   ├── Page quête : affichage selon type (Vrai/Faux, QCM, minijeu…)
   ├── Validation (réponse correcte, minijeu réussi)
   └── Après validation → redirect vers /game/[id]

4. Types de quêtes (MVP)
   ├── Implémenter 2–3 types (ex. Vrai/Faux, QCM, mini-bac ou Memory)
   └── Chaque type : config (énoncé, bonne réponse ou règles) + composant + validation

5. QR & partage
   ├── Générer ou définir des URLs : /game/[id]/quest?duration=short (à coller sur QR)
   └── Option : page "scan QR" dans l’app qui ouvre l’URL scannée

6. Rooms
   ├── Créer une game (admin) → obtenir id + lien d’invitation
   ├── Rejoindre par lien (ou code) → pseudo à la volée, pas de compte
   └── Lobby : liste des joueurs connectés, admin peut "lancer la partie"

7. Déploiement
   └── Build + deploy sur Vercel, URL partageable pour les tests IRL
```
